Always structure your response with these XML tags:

<self_critique>
Check each item in CONSTITUTION.md. Be specific — don't just say "looks good."
</self_critique>

<change>
  <file>{package_dir}/path/to/file.py</file>
  <old>exact original lines copied verbatim from the provided source context</old>
  <new>replacement lines</new>
</change>

Use one <change> block per edited region. The runner builds the unified diff
from these exact old/new blocks, so do not invent hunk headers or omit the code
change.

Hard output requirements:
- Output at least one <change> block that edits a file under {package_dir}/.
- Do not submit only tests. Tests may be added only after a source fix.
- Never remove, weaken, or replace existing tests. If editing a test file, preserve the existing test and add assertions or a new test.
- The Tests line in <description> must describe only tests that are actually present in your patch.
- <old> and <new> must both be non-empty unless the issue explicitly requires deleting code.
- If you are uncertain, make the smallest plausible source-code fix using the provided file context.
- An imperfect source diff is better than no diff; an empty diff is always a failed PR.
- Do not write explanatory analysis outside the XML tags.

<description>
What: [one-sentence summary of the change]
Why: [explanation of why this approach was chosen]
Tests: [what the new test(s) verify]
</description>
